# 后端用户体验优化点分析报告

针对当前 RAG 系统的后端架构设计，从**“保障和提升用户体验”**的角度出发，总结出以下需要优化的核心痛点及方案：

## 1. 知识库处理状态缺乏“透明度”（防盲等）
**问题分析**：
前端上传并合并分片后，后端返回成功，但实际上最耗时的**解析、分片、向量化**均在 `FileProcessConsumer` 中异步执行。
这时候用户完全不知道文件还要多久能用，甚至如果向量库由于网络原因写入失败导致任务被挂起/丢弃，用户会一直陷入“盲等”状态，发现搜不出东西后会觉得系统有 Bug。

**优化方案**：
1. **状态持久化**：在 `RagUnit` 或外层 `DocumentFile` 的数据库记录中增加 `status` 字段（如：待处理、解析中、向量化中、成功、失败）。
2. **状态感知**：后端提供一个 `/api/documents/status/{fileHash}` 轮询接口，或建立 WebSocket/SSE 通道推送实时处理进度与结果。
3. **友好异常捕获**：若重试依旧失败，记录错误原因并更新为 FAILED 状态，让终端用户能明确看到“该文件由于格式问题解析失败”。

## 2. 长期记忆（用户画像）缺乏“掌控感”
**问题分析**：
`UserProfileService` 能够聪明地在后台提炼用户的技术特征并永久保存在 Redis 中，并在后续聊天中注入给大模型。
但如果 AI **“误解”**了用户（比如错误的打上了某类标签），由于该画像对用户也是隐形的，会导致之后的所有对话都被误导。用户对此束手无策，产生“系统不受控”的反感。

**优化方案**：
1. **记忆透明化**：暴露 `GET /api/user/profile` 接口，让前端能展示现存的用户特征和偏好。
2. **开放干预权**：允许用户通过界面直接修改或删除对自己的标签描述，甚至提供一个“清除短期记忆/重置认知”的操作，消除不好的历史污染。
3. **增量提炼机制**：目前提取集中在“删除会话”时，且仅取近 20 条。在长对话场景中容易丢失前面的核心背景。可以优化为按流或固定轮次（每 10 轮）异步增量更新画像。

## 3. 检索配置过于死板，无法满足“发散 vs 严谨”的动态需求
**问题分析**：
`RagRetrievalService` 强依赖于配置文件中硬编码的 `similarity-threshold` 和 `hit-score-threshold`。
这导致检索尺度一刀切。有时用户需要严格按文档回答（严谨），有时需要大模型结合自身的智慧拓展回答（发散）。

**优化方案**：
1. **按需调节参数**：通过 `MultiTurnChatRequest` 参数接受前端下发的偏好度控制参数（如 `strictness` 或 `creativity`）。
2. **动态调整阈值**：后端根据此参数动态调节 Rerank 的及格线，或者在组装给大模型的 Prompt 中动态变更 “如未查到必须回答不知道” 的严厉限制程度。

## 4. 全局异常处理对用户“不够友好”
**问题分析**：
多个 Controller 接口在 `catch (Exception e)` 中直接暴力的 `return ApiResponse.serverError(e.getMessage())`。
如果出现底层报错（例如 `NullPointerException` 或数据库连接异常），前端弹出的错误往往是英文或长串代码。

**优化方案**：
1. **全局统一拦截**：使用 `@RestControllerAdvice` 统一接管未捕获的运行时异常。
2. **友好的兜底提示**：将非业务性、非用户视角的代码报错，统一转换为诸如“**服务器繁忙，处理超时，请稍后再试**”的柔性提示语。

## 5. 断点续传下的“孤儿分片”占用资源
**问题分析**：
`ChunkUploadController` 的分片上传机制很好，但如果用户传到一半不再继续，分片长期堆积在 `file.upload.temp-dir` 或 MinIO 临时区。
长此以往不仅浪费资源，在做 C 端配额管理时还会计算混乱。

**优化方案**：
1. **自动清理策略**：添加 `@Scheduled` 定时清理任务，每夜扫描并回收超过一定时间（如 48 小时）未完成合并的分片数据和 Redis 状态记录。
